### Priority 3 — double-submit idempotency (commit `3c5aeff`) · frontend only

The Denkmal answer rendered twice (You·20:46 + You·20:49, identical). Not a React double-render
(the 3-minute gap ruled that out). Root cause: `useChatTurn.onMutate` and `.mutationFn` each ran
`input.clientRequestId ?? crypto.randomUUID()` **independently**, and `handleSubmit` passed no id —
so a single submit minted two different uuids. The optimistic placeholder (and the id the Recovery
banner reads off the failed turn) used `uuidB` while the server persisted `uuidA`; the qualifier
error (visible in the walk's 50%-error Logs) then drove a Retry that re-submitted `uuidB`, which the
server's `(project_id, client_request_id, role)` unique index did **not** see as the same turn → a
second user row was persisted, and `useMessages` (whole-list refetch) rendered both. Fix: one stable
`clientRequestId` per turn (`handleSubmit` mints it; `onMutate` resolves it onto the shared input so
`mutationFn` reuses it; optimistic write is now idempotent by `client_request_id`). The **existing**
server idempotency then dedupes the retry — no SQL/edge-fn change. Deliberately **not** a
content+time-window dedupe (that would wrongly drop a legitimately repeated answer). Test:
`scripts/smoke-double-submit.mts` (8 assertions) + smokeWalk source guards.
